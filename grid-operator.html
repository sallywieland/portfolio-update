Case Study · Native App · Energy Infrastructure

Real-time data for a grid that never stops.

A regional grid operator serving 42 million people needed a native mobile app for operators and market participants to monitor real-time grid conditions. 303,800 data points. Updates every 5 seconds. The margin for confusion is zero.

Client

Regional grid operator (confidential)

Platform

Native mobile (iOS + Android)

Role

Sole designer, start to handoff

Market scale

~$2B settled per month

Real-time displays: generation fuel mix, locational marginal pricing, ACE charts, ancillary market, schedule interface.

The problem

Critical infrastructure. A 5-second refresh cycle. No room to misread anything.

The operator's public data system publishes over 300,000 data points updated every 5 seconds across 15 states. Their existing tooling was desktop-only, built for control rooms. Market participants needed the same situational awareness on the floor, in the field, on the go.

This meant dense data displays that had to work at a glance, in adverse conditions, without ambiguity. No sparklines-as-decoration. No hiding complexity. Just making it legible.

Constraints

The hard parts were the interesting parts.

Energy infrastructure design isn't a visual problem. The visual layer is the last 10%. The first 90% is understanding what a transmission operator actually needs at 2am when an event occurs, and what a market participant needs on the trading floor when LMPs spike.

Data density

Charts needed to convey meaningful signal at small sizes on mobile, without simplifying to the point of uselessness. Every chart element had to earn its space.

Real-time refresh

5-second updates across dozens of data series. The UI had to update without disorienting the user. Values change; context shouldn't flicker.

Dual audiences

Transmission operators and market participants have different mental models for the same data. One display had to serve both without becoming a menu of settings.

Regulatory vocabulary

LMP, ACE, ancillary markets: terms the users know cold. The app had to use the right language without explanation, not consumer-friendly softening.

Design decisions

What I chose and why.

Dark UI for the data, not the aesthetic

Dark backgrounds improve contrast for chart data in variable lighting conditions. Control room environments and trading floors aren't always brightly lit, and glare on screens is a real operational issue. The dark theme was a functional call.

Consistent chart geometry across every view

Same axis conventions, same color encoding, same scale behavior across all screens. When a user scans between generation fuel mix and LMP pricing, visual consistency reduces the cognitive load of re-orienting. They're reading data, not learning a new chart each time.

No hiding the density

The instinct with dense B2B data is to progressively disclose, hide behind toggles, simplify to summary views. That's the wrong call here. These users are professionals who came for the data. Hiding it wastes their time. The job was to organize it, not reduce it.

Real content from day one

I used Claude to generate realistic data ranges for every chart type before any layout was finalized. LMP values with realistic variance across nodes. Generation fuel mix percentages that actually summed to 100. ACE values that reflected typical operating ranges. Lorem ipsum doesn't break mobile charts. Real numbers do.
